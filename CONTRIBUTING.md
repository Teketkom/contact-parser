## 2. Как начать

### 2.1 Форкинг репозитория

```bash
# 1. Форкнуть репозиторий через GitHub UI
# 2. Клонировать форк
git clone https://github.com/YOUR-USERNAME/contact-parser.git
cd contact-parser

# 3. Добавить upstream
git remote add upstream https://github.com/your-org/contact-parser.git

# 4. Убедиться, что remote настроен правильно
git remote -v
```

### 2.2 Настройка окружения разработки

```bash
# Backend
cd backend
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
pip install -r requirements-dev.txt
playwright install chromium

# Frontend
cd ../frontend
npm install

# Pre-commit хуки
cd ..
pip install pre-commit
pre-commit install

# Настроить .env
cp .env.example .env
# Отредактировать .env: добавить API-ключи
```

### 2.3 Проверка, что всё работает

```bash
# Backend тесты
cd backend
pytest -m "unit" -v

# Frontend тесты
cd ../frontend
npm test

# Docker (опционально)
docker compose up -d
curl http://localhost:8000/health
```

## 3. Процесс разработки

### 3.1 Стратегия ветвления

Проект использует **упрощённый Git Flow**:

| Ветка | Назначение | Правила |
|---|---|---|
| `main` | Production-ready код | Только merge из `release/*` и `hotfix/*` |
| `develop` | Интеграционная ветка | Merge из `feature/*` и `fix/*` |
| `feature/TASK-XXX-description` | Новая функциональность | Создаётся от `develop` |
| `fix/TASK-XXX-description` | Исправление ошибки | Создаётся от `develop` |
| `hotfix/vX.Y.Z` | Критическое исправление в prod | Создаётся от `main` |
| `release/vX.Y.Z` | Подготовка релиза | Создаётся от `develop` |
| `chore/description` | Технические задачи | Создаётся от `develop` |
| `docs/description` | Только документация | Создаётся от `develop` |

### 3.2 Создание рабочей ветки

```bash
# Синхронизировать с upstream
git fetch upstream
git checkout develop
git merge upstream/develop

# Создать ветку для функции
git checkout -b feature/TASK-42-add-kpp-validation

# Создать ветку для исправления
git checkout -b fix/TASK-43-phone-regex-mobile
```

### 3.3 Конвенция коммитов

Проект следует стандарту [Conventional Commits](https://www.conventionalcommits.org/).

**Формат:**
```
<тип>(<область>): <краткое описание>

[необязательное тело коммита]

[необязательный футер]
```

**Типы:**

| Тип | Использование |
|---|---|
| `feat` | Новая функциональность |
| `fix` | Исправление ошибки |
| `docs` | Изменения в документации |
| `refactor` | Рефакторинг (не fix, не feat) |
| `test` | Добавление или изменение тестов |
| `chore` | Технические задачи (зависимости, CI, конфиги) |
| `perf` | Улучшение производительности |
| `ci` | Изменения CI/CD конфигурации |
| `revert` | Откат изменений |

**Примеры хороших коммитов:**
```
feat(extractor): добавить извлечение КПП через LLM в Variant B

fix(crawler): исправить таймаут при обработке SPA-сайтов с React Router

docs(api): обновить примеры curl для endpoint POST /tasks

test(extractor): добавить параметризованные тесты форматов телефонов

chore(deps): обновить playwright с 1.41.0 до 1.42.0

refactor(task_manager): заменить threading.Lock на asyncio.Lock
```

**Примеры плохих коммитов:**
```
❌ fix
❌ Update
❌ Исправил баг
❌ WIP
❌ asdfasdf
```

## 4. Стандарты кода

### 4.1 Python (backend)

**Форматирование:**
```bash
# Запустить форматтер (обязательно перед коммитом)
ruff format backend/

# Запустить линтер
ruff check backend/

# Проверка типов
mypy backend/
```

**Требования:**
- Python 3.11+ синтаксис
- Строгая типизация: все публичные функции должны иметь аннотации типов
- Docstrings для всех публичных классов, методов и функций (Google style)
- Длина строки: максимум 100 символов
- Без неиспользуемых импортов

**Пример правильно оформленной функции:**
```python
async def extract_fio(
    html_fragment: str,
    *,
    min_confidence: float = 0.7,
) -> list[tuple[str, float]]:
    """Извлечь ФИО из HTML-фрагмента.

    Args:
        html_fragment: Строка с HTML-контентом.
        min_confidence: Минимальный порог уверенности (0.0–1.0).

    Returns:
        Список кортежей (ФИО, уверенность).

    Raises:
        ExtractionError: При критической ошибке обработки HTML.

    Example:
        >>> results = await extract_fio("<p>Иванов Иван Иванович</p>")
        >>> assert results[0][0] == "Иванов Иван Иванович"
    """
    ...
```

### 4.2 TypeScript (frontend)

**Форматирование:**
```bash
cd frontend
npm run lint    # ESLint
npm run format  # Prettier
```

**Требования:**
- Только функциональные компоненты (class components запрещены)
- Явная типизация props (interface Props, не any)
- Использование React хуков (useState, useEffect, useCallback)
- Tailwind CSS для стилизации (никаких inline-стилей без крайней необходимости)
- Компоненты в PascalCase, файлы в PascalCase.tsx

## 5. Тестирование

### 5.1 Обязательные требования

- **Каждая новая функция** должна иметь хотя бы один unit-тест
- **Каждый исправленный баг** должен иметь регрессионный тест
- **Покрытие кода:** не должно снижаться ниже 70%
- **Все существующие тесты** должны проходить

### 5.2 Запуск тестов

```bash
cd backend

# Быстрая проверка (перед коммитом)
pytest -m "unit" -q

# Полная проверка (перед PR)
pytest --cov=app --cov-report=term-missing -v

# Проверка метрик качества
python tests/quality/measure_metrics.py
```

### 5.3 Именование тестов

```python
# Шаблон: test_[что тестируется]_[при каких условиях]_[ожидаемое поведение]

def test_extract_email_with_mailto_link_returns_normalized():
    ...

def test_create_task_without_urls_returns_422():
    ...

def test_robots_txt_disallow_skips_url():
    ...
```

## 6. Создание Pull Request

### 6.1 Перед созданием PR

```bash
# 1. Синхронизировать с upstream/develop
git fetch upstream
git rebase upstream/develop

# 2. Запустить все проверки
cd backend
ruff check .
mypy .
pytest --cov=app -q

cd ../frontend
npm run lint
npm test

# 3. Убедиться, что нет лишних файлов
git status
# В .gitignore должны быть: .env, __pycache__, .venv, node_modules
```

### 6.2 Шаблон описания PR

При создании Pull Request заполните все разделы:

```markdown
## Тип изменений
- [ ] feat: Новая функциональность
- [ ] fix: Исправление ошибки
- [ ] refactor: Рефакторинг
- [ ] docs: Только документация
- [ ] chore: Технические задачи

## Описание
[Опишите, что сделано и зачем]

## Связанные задачи
Closes #[номер Issue]

## Изменения
- [ ] Добавлен/изменён файл X
- [ ] Обновлена функция Y
- [ ] Добавлен модуль Z

## Тестирование
- [ ] Написаны unit-тесты для новой функциональности
- [ ] Все существующие тесты проходят (`pytest`)
- [ ] Линтер не выдаёт ошибок (`ruff check`)
- [ ] Типы корректны (`mypy`)
- [ ] Проверены метрики Precision/Recall (если затронут экстрактор)

## Чеклист
- [ ] Код соответствует стайл-гайду
- [ ] Документация обновлена (если нужно)
- [ ] CHANGELOG.md обновлён
- [ ] Нет хардкоженных API-ключей или паролей
- [ ] robots.txt-совместимость не нарушена

## Скриншоты (если UI-изменения)
[Добавьте скриншоты до/после]
```

### 6.3 Процесс code review

1. PR назначается разработчику-ревьюеру
2. Ревьюер проверяет код в течение 48 часов
3. Замечания вносятся через GitHub Review Comments
4. Автор PR исправляет замечания и отвечает на каждое
5. После одобрения — Squash Merge в `develop`
